# Asset Types Reference

Types of champion enablement materials.

## ROI Model

### Purpose
Quantify financial impact to justify investment.

### When to Use
- CFO/Finance needs to approve
- Budget justification required
- ROI skepticism present
- Business case needed

### Components
- Current state costs
- Projected savings
- Implementation investment
- Timeline to value
- ROI calculation
- Payback period

### Tips
- Use their actual numbers
- Be conservative (under-promise)
- Include sensitivity analysis
- Show multiple scenarios

### Format
Spreadsheet or markdown with tables.

---

## Executive One-Pager

### Purpose
Give champion a concise summary to share with executives.

### When to Use
- Champion needs to brief executives
- Limited exec attention time
- Need to "leave behind"
- Internal advocacy

### Components
- Problem summary (one sentence)
- Solution overview
- Expected results (3-4 metrics)
- Investment and timeline
- Call to action

### Tips
- One page maximum
- Lead with their problem
- Numbers, not adjectives
- Clear next steps

### Format
Single page PDF or markdown.

---

## Objection Responses

### Purpose
Arm champion to counter specific concerns.

### When to Use
- Known objections from stakeholders
- Blocker with specific concern
- Champion facing pushback
- Internal FAQ needed

### Components
- Objection stated clearly
- Short response (2-3 sentences)
- Supporting evidence
- Proof point/example

### Tips
- Address their specific words
- Don't be defensive
- Provide evidence
- Keep responses concise

### Format
FAQ-style document.

### Example

```markdown
## Objection: "This is too expensive"

**Response:**
The investment of [$X] pays for itself in [Y months] through ticket
automation savings of [$Z/month]. Compared to hiring additional
support agents at [$cost/agent], this is [X%] less expensive while
handling [Y%] more volume.

**Evidence:**
- [Customer Name] saw [X%] ROI in first year
- Our average customer payback period is [Y months]
```

## Business Case

### Purpose
Formal justification document for committee/board.

### When to Use
- Formal approval process
- Multiple stakeholders to convince
- Large investment
- Strategic decision

### Components
- Executive summary
- Problem statement
- Solution overview
- Market analysis
- Financial analysis
- Implementation plan
- Risk analysis
- Recommendation

### Tips
- Follow their template if they have one
- Comprehensive but readable
- Address all stakeholder concerns
- Include alternatives considered

### Format
Multi-page document with sections.

---

## Competitive Comparison

### Purpose
Show differentiation vs. alternatives.

### When to Use
- Active competitive evaluation
- Champion asked "why you vs X"
- Need to counter competitor claims

### Components
- Feature comparison table
- Unique differentiators
- Customer proof points
- Honest assessment (including where competitor wins)

### Tips
- Be honest about competitor strengths
- Focus on what matters to them
- Use their evaluation criteria
- Provide evidence, not claims

### Format
Comparison table with narrative.

---

## Technical Summary

### Purpose
Address technical stakeholder concerns.

### When to Use
- CTO/IT evaluation
- Architecture review
- Integration discussion
- Security assessment

### Components
- Architecture overview
- Integration approach
- Security/compliance
- Scalability
- Implementation requirements

### Tips
- Match their technical depth
- Address their specific stack
- Include diagrams
- Reference documentation

### Format
Technical document with diagrams.

---

## Implementation Plan

### Purpose
Show what implementation looks like.

### When to Use
- Timeline questions
- Resource planning
- Reducing implementation fear
- Setting expectations

### Components
- Phase breakdown
- Timeline with milestones
- Resource requirements (both sides)
- Dependencies
- Success criteria per phase

### Tips
- Be realistic, not optimistic
- Show quick wins
- Identify their responsibilities
- Include contingencies

### Format
Timeline/Gantt or phased document.

---

## Case Study Summary

### Purpose
Social proof from similar customer.

### When to Use
- "Has anyone done this before?"
- Need credibility
- Skepticism about results
- Reference request

### Components
- Customer profile (similar to them)
- Challenge (similar to theirs)
- Solution (what we did)
- Results (quantified)
- Quote from customer

### Tips
- Match industry/size if possible
- Use specific numbers
- Include before/after
- Offer reference call

### Format
1-2 page case study.

---

## Asset Selection Guide

| Situation | Recommended Asset |
|-----------|-------------------|
| CFO needs to approve | ROI Model |
| CEO briefing | Executive One-Pager |
| Finance committee | Business Case |
| Technical evaluation | Technical Summary |
| Competitive evaluation | Competitive Comparison |
| Specific objection | Objection Responses |
| Timeline questions | Implementation Plan |
| Need social proof | Case Study Summary |

---

## Asset Quality Standards

### Must Have
- Company name personalized
- Their numbers (not generic)
- Their pain points
- Professional formatting
- Easy to share/present

### Should Have
- Relevant proof points
- Visual elements (charts, tables)
- Clear call to action
- Contact information

### Avoid
- Generic templates
- Overly promotional language
- Unsubstantiated claims
- Walls of text
- Technical jargon (unless for technical audience)
